## 1. 概要

駐車場の入出庫をAIカメラで検知してカウントし、現在の駐車台数から
「満（満車）／空（空車）／混（混雑）」のステータスを判定する。
最終的にはステータスをLEDサイネージへ出力して掲示する。
Raspberry Pi 上で動作するエッジシステムとして構築する。

## 2. 登場人物 / 役割分担

| 主体 | 担当範囲 |
| --- | --- |
| クライアント | AIカメラ本体・LEDサイネージ本体の用意（機種は既定 or 選定中）。設置場所・運用条件。 |
| ハードウェア開発（外部業者） | 屋外配線、屋外設置用の筐体・ボタン（手動カウント補正用デバイス）の製作・設置。 |
| ソフトウェア開発（自分） | Raspberry Pi 上のソフト一式。カメラ信号の受信、カウント、状態保持・判定、（後に）サイネージ出力。 |

## 3. スコープ（フェーズ分け）

### フェーズ1（= 今回の第一目標）
- **カメラからの信号を Raspberry Pi で受け取る。**
- **Pi 内部で現在台数をカウントする（入庫 +1 / 出庫 -1）。**
- **状態（現在台数・満空混ステータス）を保持する（電源断後も復元できる）。**
- 動作確認できる手段（ログ出力／簡易な状態確認）を用意する。
- ※ サイネージ出力・手動ボタン・LinkBase連携はこの段階では作り込まない（後述のとおり拡張しやすい形にだけしておく）。

### フェーズ2以降（今回スコープ外、設計上の余地は残す）
- LEDサイネージへの出力。
- 手動カウント補正（外部業者製ボタンからの接点入力）。
- LinkBase 連携（外部送信）。
- Web画面等での監視・操作。

| 用語 | 意味 |
| --- | --- |
| 接点 / ドライ接点 | 無電圧接点。リレー等の機械的なON/OFF。電圧を持たず、受け側でプルアップして検出する。 |
| 満空混ステータス | 駐車場の混雑状態を表す3値（満 / 空 / 混）。 |

## 5. 機能要件

### 5.1 カメラ信号の受信（フェーズ1）
- AIカメラ（Hanwha Vision XNO-A6084R）からの「入庫」「出庫」イベントを受信する。
- 受信方式は **(A) GPIO 接点入力で確定**（カメラのアラーム出力 IO1/IO2 → フォトカプラ等 → Pi の GPIO）。
  ネットワーク経由（ONVIF/SUNAPI/HTTP）は **使わない**。受信層は差し替え可能な作りにしておく（保険・テスト容易性のため。ネットワーク実装はスタブのみ）。
- カメラ側はAIエンジンの「仮想線（交差・方向）」または「車両カウント」イベントを、IO1 を入庫、IO2 を出庫として **アラーム出力に紐付ける** 設定で運用する（カメラ Webviewer で設定。設定者は別途決める）。
- カメラのアラーム出力は **オープンコレクタ方式**（無電圧接点ではない）。Pi 側は内蔵プルアップ（あるいはフォトカプラ経由）で受け、出力 ON 時に LOW として検出する。
- ソフト内部は「入庫イベント」「出庫イベント」の **2種類の抽象イベント** として扱う。
- **本システムの責務は「入庫信号」「出庫信号」の2系統を処理することのみ。** カメラの設置・調整、および複数カメラ／複数ゲートがある場合に各イベントを「入庫線」「出庫線」へ集約することは、クライアント（カメラ）／外部業者（配線）側の責務とする。ソフトから見える入力は常に2系統。
- チャタリング／重複通知対策（デバウンス、最小イベント間隔）を行う。

### 5.2 カウント（フェーズ1）
- 入庫イベントで現在台数 +1、出庫イベントで -1。
- 下限 0、上限 = 総台数 でクランプ（範囲外イベントはログに警告を残す）。
- 総台数は設定ファイルで指定（暫定値で開始してよい）。

### 5.3 状態の保持・永続化（フェーズ1）
- 現在台数・最終更新時刻・現在ステータスを永続化（SQLite もしくはファイル）。
- 起動時に直近の状態を復元する。
- 入出庫イベント／ステータス変化／エラーをローカルにログとして記録する。

### 5.4 満空混判定（フェーズ1で内部判定まで、出力はフェーズ2）
- **残台数の割合（残台数 / 総台数）** で判定する。
  - **満**: 残台数の割合 ≤ 下限しきい値（暫定: 0% = 残0台）
  - **混**: 下限しきい値 < 残台数の割合 ≤ 中間しきい値（暫定: 0%超〜10%）
  - **空**: 残台数の割合 > 中間しきい値（暫定: 10%超）
- 総台数・しきい値は設定ファイルで変更可能。
- 状態が変化したときのみ確定（境界付近のバタつき防止にヒステリシスを検討）。
- ※ 具体的な総台数・% はクライアントの感覚値が出るまで暫定値で進める。

### 5.5 手動カウント補正（フェーズ2）
- 補正用ボタンは **外部業者が屋外設置用に製作**。本システムは「汎用のデジタル接点入力（+1用 / -1用、必要なら リセット用）」を受ける口を用意する。
- ボタン⇔Pi の I/F 仕様（電圧、プルアップ方向、コネクタ、パルス幅、チャタリング条件）は外部業者と合意のうえ別途仕様書化する。
- 補正操作は履歴として記録する。

### 5.6 LEDサイネージ出力（フェーズ2）
- 判定結果（満 / 空 / 混）を出力する。出力方式（接点 / シリアル等）はサイネージ機種に合わせて決める。

## 6. 非機能要件

### 6.1 ハードウェア構成
- コントローラ: Raspberry Pi（モデルは §8.3 で確定。GPIO 使用想定 / 暫定 Pi 4 系）。
- 入力: **Hanwha Vision XNO-A6084R**（2MP AI IRバレットカメラ）。給電は PoE (IEEE802.3af Class3, 最大 10.2W)、Ethernet 10/100BASE-T (RJ-45)。アラーム I/O ケーブル: オレンジ=IO1, 茶=IO2, 黒=GND。出力は **オープンコレクタ**。動作温度 -40〜+60℃、IP66/IP67、IK10、NEMA TS-2 で屋外設置可。
- 出力（フェーズ2）: LEDサイネージ。
- カメラ OC 出力を GPIO で受けるため、フォトカプラ（推奨）または抵抗分圧で電圧整合・絶縁する。GND は共通化。屋外配線・サージ対策・PoEインジェクタ／スイッチの手配は外部業者と分担。

### 6.2 信頼性
- 電源断後の自動復旧（systemd 自動起動 + 状態の永続化からの復元）。
- 接点ノイズ・誤検知・重複通知への耐性（デバウンス、範囲外検出、ログ）。

### 6.3 運用・保守
- 総台数・しきい値・GPIO ピン割当・受信方式などは設定ファイルで変更可能。
- ログから入出庫履歴とステータス変化を追える。

### 6.4 ソフトウェア
- 言語: **Python 3**（gpiozero でGPIO制御。実機が無い環境では mock pin factory / ダミー受信層で動かす）。
- 構成: 「イベント受信層（GPIO実装＋差し替え用スタブ）」→「カウント・状態管理（コア）」→「永続化／ログ」→（フェーズ2で）「出力層」。
- 自動起動: systemd サービス化。
- 詳細・実装手順は §10。

### 8.1 カメラ（確認結果反映済 / 2026-05-13）

**確定: Hanwha Vision XNO-A6084R**（2MP AI IRバレットカメラ。データシートと取扱説明書PDFは `docs/` に格納）。

**確認結果（PDFより）:**
- ✅ **アラーム出力端子: あり（2 configurable I/O ports）**。入出力共用で、Webviewer 上で各ポートを入力 or 出力に個別設定可能。ケーブル配色: オレンジ=IO1, 茶=IO2, 黒=GND（提供されたオーディオ/アラームケーブル経由）。

- ✅ **トリガ条件: AIエンジンで「仮想線（交差・方向）」「IVA(進入/退出, 方向)」「車両カウント」が分析イベントとして利用可。** これらをアラーム出力に紐付け可能（Alarm Events に「アラーム出力」を含む）。

- ❓ **出力のパルス幅／設定可否**: マニュアル本文に明記なし。Webviewer のアラーム出力設定で持続時間を指定できるのが Hanwha 共通仕様 → 実機セットアップ時に確認し、`bounce_time` / `min_interval` の調整に反映する。
- 参考: ネットワーク経由のイベントも可能（ONVIF: s/G/T/M、SUNAPI HTTP API、MQTT publication）。**今回は使わない方針** だが、もし将来 OC 配線が困難になった場合の代替手段として認識しておく。

### 10.3 設定ファイル（config.toml）スキーマ案
```toml
[parking]
total_spaces = 100            # 総駐車台数（暫定値。実値はクライアント確認待ち）

[thresholds]
# 残台数の割合（残/総）で判定。full_ratio 以下=満、crowded_ratio 以下=混、それ超=空
full_ratio    = 0.0           # 暫定: 残0台で満
crowded_ratio = 0.10          # 暫定: 残10%以下で混
hysteresis    = 0.0           # 境界のバタつき防止のマージン（0で無効。実装は任意で後回し可）

[receiver]
type = "gpio"                 # "gpio" | "dummy"
[receiver.gpio]
entry_pin   = 17              # 入庫接点を受けるBCMピン（暫定。実配線確定後に更新）
exit_pin    = 27              # 出庫接点を受けるBCMピン
pull_up     = true            # 接点はGND短絡でON想定 → 内部プルアップ
bounce_time = 0.05            # デバウンス秒
min_interval = 0.2            # 同一方向の連続イベントを無視する最小間隔（秒）
active_state_pulse = true     # パルス（立ち下がり1回=1イベント）として扱う

[storage]
db_path = "parking.db"

[logging]
level = "INFO"
file  = "parking.log"
```

### 10.4 コアの振る舞い（counter.py）
- `current` を保持。`record_entry()` → `current = min(current+1, total)`、`record_exit()` → `current = max(current-1, 0)`。
- クランプで弾いた（範囲外）イベントは戻り値 or 例外で呼び出し側に伝え、WARNING ログを残す。
- `status()` → 残台数 `total - current` の割合で `FULL / CROWDED / EMPTY` を返す。
  - `remaining_ratio <= full_ratio` → FULL、`<= crowded_ratio` → CROWDED、それ以外 → EMPTY。
  - ヒステリシス: 直前ステータスからの遷移時にしきい値を少しずらす（実装は任意。まずは無しでOK、設定で後付け）。
- 状態が変化したら `store` に永続化し、ステータス変化を INFO ログ＋events テーブルに記録。

### 10.5 永続化（store.py）
- `state` テーブル: `current_count INTEGER, status TEXT, updated_at TEXT`（1行のみ upsert）。
- `events` テーブル: `id, ts, type('entry'|'exit'|'status_change'|'error'), detail`。
- 起動時 `restore()` で `state` を読んで counter に流し込む。テーブルが空なら `current=0` で初期化。

### 10.6 受信層（receivers/）
- `EventReceiver(base)`: `start()` / `stop()`、コンストラクタで `on_entry`, `on_exit` コールバックを受ける。
- `gpio.py`: `gpiozero.Button(entry_pin, pull_up=..., bounce_time=...)` を2つ。`when_pressed`（または when_released）で `on_entry`/`on_exit` を呼ぶ。`min_interval` で連発を抑制。実機が無い環境では `gpiozero` の `Device.pin_factory = MockFactory()` で代用できるようにしておく。
- `dummy.py`: 開発・E2E確認用。`stdin` から `i`/`o` を読む、または小さな HTTP サーバ（`POST /entry` `/exit`）、または周期的にランダム発火、のいずれか実装（まずは stdin が手軽）。`config.receiver.type = "dummy"` で切替。

### 10.7 アプリ配線（app.py / __main__.py）
1. `config.toml` ロード（無ければ `config.example.toml` をコピーする旨を案内）。
2. logging 初期化。
3. `store` 初期化 → `restore()` → `counter` 構築。
4. `config.receiver.type` で受信層を選んでインスタンス化。`on_entry = lambda: handle(counter.record_entry)`、`on_exit` 同様。
5. `handle`: counter 更新 → 範囲外なら WARN → status 再計算 → 変化あれば store 更新＋ログ。
6. `receiver.start()` してイベントループ / シグナル待ち（SIGTERM で `stop()`＋flush）。

### 10.8 動作確認（実機なしでできる範囲）
- `config.toml` の `receiver.type = "dummy"` にして `python -m parking` 起動。
- stdin に `i`（入庫）/`o`（出庫）を打ち込み、ログとDBの `current_count`・`status` が想定どおり変化するか確認。
- 一度落として再起動 → 直前の `current_count` が復元されるか確認（永続化テスト）。
- 0未満／総台数超のイベントを打って WARN が出る＆値が動かないことを確認。

### 10.9 実機投入時にやること（カメラ・Pi確定後）
- `config.toml` の `receiver.type = "gpio"`、`entry_pin`/`exit_pin`/`pull_up`/`bounce_time` を実配線に合わせて設定。
- **カメラ (XNO-A6084R) 側設定**: Webviewer で IO1/IO2 を「出力」に設定。AIエンジンの「仮想線(交差・方向)」または「車両カウント」イベントを、入庫→IO1、出庫→IO2 のアラーム出力に紐付ける（クライアント／設置担当の作業。こちらは設定手順を指示）。出力パルス幅に合わせて `bounce_time`・`min_interval` を調整。
- **Pi 側配線**: カメラ OC 出力（IO1, IO2）→ フォトカプラ → Pi GPIO（内蔵プルアップ）。GND を共通化。OC許容V/Aの実値が出たらフォトカプラ駆動側の電流制限抵抗を確定。
- `systemd/parking.service` を `/etc/systemd/system/` に置いて `enable --now`。
- カメラ・GPIO 間のフォトカプラ等の結線は外部業者と分担（§6.1）。
